# Notification System Design

## Architecture Overview
The notification system is designed to handle asynchronous event processing to notify mechanics and drivers about optimal maintenance schedules.

## Components
1. **API Gateway / Event Listener**: Receives schedules generated by the Vehicle Maintenance Scheduler.
2. **Message Queue (e.g., RabbitMQ / Kafka / Redis Streams)**: Buffers incoming notifications.
3. **Notification Workers**: Processes messages from the queue and formats them for the target platform (SMS, Email, Push).
4. **Delivery Providers**: SendGrid (Email), Twilio (SMS).

## Data Flow
1. The `vehicle_maintenance_scheduler` completes a scheduling cycle and publishes a `ScheduleGenerated` event.
2. The `notification_app_be` consumes this event.
3. The system maps the Depot and Vehicle IDs to actual user communication preferences.
4. Notifications are dispatched via configured delivery providers.
